# CSC-113 AI Fundamentals: Complete Course Outline
## Module-by-Module Guide with Assignment Status

**Course**: CSC-113 Artificial Intelligence Fundamentals  
**Credits**: 3 (2 hours class + 2 hours lab)  
**Prerequisites**: None  
**Target Audience**: Ages 18-65, mixed technical abilities  

---

## COURSE OVERVIEW

### Official Description
"This course provides a survey of artificial intelligence and machine learning. Topics include the history, development, and current applications of artificial intelligence and machine learning. Upon completion, students should be able to demonstrate general artificial intelligence and machine learning concepts."

### Course Philosophy
**"We're not teaching them to code. We're teaching them to learn."**

This course uses AI tools as the context for teaching second-order skills:
- **Documentation** - Clear communication of process and results
- **Resilience** - "Failure is just exercise" - iteration over perfection
- **Growth Mindset** - Learning from mistakes, celebrating progress
- **Critical Thinking** - Questioning AI outputs, understanding limitations
- **Professional Workflow** - GitHub-first from day one

### Microcredential Badge
Students completing this course earn a **Prompt Engineering & AI Collaboration** badge demonstrating:
- Effective AI tool usage across multiple platforms
- Systematic prompt engineering techniques
- Critical evaluation of AI outputs
- Professional documentation practices
- Portfolio-ready AI collaboration projects

## WEEK 5: ITERATIVE IMPROVEMENT - "THE GOOD(ISH) BOT"
**Theme: "One Small Fix, Big Learning"**

### Learning Objectives
- Practice systematic iteration
- Understand trade-offs in design decisions
- Test before/after comparisons
- Document incremental improvement
- Recognize that "fixed" often means "different trade-offs"

### Topics Covered
- Incremental development methodology
- A/B testing basics
- Trade-off analysis
- When to iterate vs. rebuild

**Assignment 4.1: "The Good(ish) Bot"**
- **Status**: üü¢ 85% Complete (full assignment + rubric exists)
- **Needs**: Example submissions, GitHub structure

**What Students Do:**
1. Create Issue #4: "Improve [Bot Name] - Single Change"
2. Save 3 test cases from Bad Bot as baseline
3. Identify THE problem (not all problems)
4. Make EXACTLY ONE change to system prompt
5. Run same test cases, compare results
6. Document trade-offs of the change
7. Reflect on what's still broken (on purpose)

**The One-Change Rule:**
Students cannot fix everything. They must:
- Choose the most impactful friction point
- Make minimal change targeting that issue
- Keep bot's personality recognizable
- Accept that new problems may emerge

**Example Improvements:**
- **MANSPLAINBOT** ‚Üí Adds citations (still condescending, but now verifiable)
- **OFFBYONE** ‚Üí Adds tiny footnotes with correct numbers (still wrong, but transparent)
- **KEVIN JR.** ‚Üí Actually gives answer first, THEN suggests meeting (helpful, still corporate)

**Deliverable:**
- Original bot (v1) preserved
- Improved bot (v2) with documentation of change
- Before/after test comparison table
- Trade-off analysis
- Reflection on iteration process

## INSTRUCTOR IMPLEMENTATION NOTES

### First Time Teaching This Course

**Month Before:**
- Complete GitHub organization setup
- Recruit community partners
- Prepare Week 1-2 materials completely
- Test all AI tools and rate limits

**Week 1:**
- Set expectations clearly
- GitHub workflow demonstration
- "Failure is exercise" culture establishment
- Daily standup ritual begins

**Ongoing:**
- Review GitHub activity daily (5 min)
- Approve PRs within 24 hours
- Intervene when students stuck 3+ days
- Maintain "Kevin from IT" persona

**Common Pitfalls:**
- Don't teach Git commands first - web interface only initially
- Don't let students commit to main - protection from day 1
- Don't skip daily standups - ritual is important
- Don't panic when AI tools change - teaching adaptability

### Budget Considerations

**Free Tier Management:**
- Gemini Flash cascade (Flash-Lite ‚Üí Flash ‚Üí Pro)
- 900 requests/day/student target (leaves buffer)
- Monitor rate limits Weekly
- Have backup tools ready

**GitHub Education:**
